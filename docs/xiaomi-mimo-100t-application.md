# Xiaomi MiMo 100T 申请内容草稿

## 04 使用 Agent 或 AI 驱动构建的具体成果

我构建并长期运行了 Genesis，一个面向 QQ 私聊和群聊场景的 AI Agent 系统。它要解决的核心痛点是：普通聊天机器人通常只能被动回复、工具调用依赖关键词、复杂任务容易漏步骤或重复执行，且在群聊中很难判断“该不该插话”。Genesis 的目标是把机器人从单轮问答升级为可感知上下文、可规划任务、可调用工具、可维护自身能力的 Agent。

核心逻辑流分为几层：第一层是 Sentry 哨兵，持续监听群聊和私聊，根据是否被 @、是否提到机器人、问题强度、连续追问、群聊热度、冷却时间等信号计算响应意愿，必要时再调用 LLM 做最终判断，避免机器人刷屏或漏掉重要请求。第二层是 Router 规划器，把用户消息转成结构化 TaskPlan，判断是否需要工具、需要哪些步骤，以及步骤之间是否有依赖。例如用户让机器人“画 @某人的头像风格图”，Router 会规划为：获取头像 -> 调用视觉模型识别头像内容 -> 把头像作为参考图交给绘图工具生成图片。第三层是 Tech / Orchestrator 执行层，按依赖拓扑顺序或并行模式调用工具，支持参数占位符传递、任务缓存、超时、取消、执行日志和结果聚合。第四层是 Persona 回复层，把工具结果转成符合机器人设定的自然回复，并结合用户画像、好感度、历史记忆和情绪分析调整语气。

目前项目已经接入 30 多个工具模块，包括绘图、图生图、识图、头像获取、网页研究、天气、音乐、文件/音视频读取、博客发布、定时任务、群管理、任务状态查询、工具日志和技能创建/管理等。工具系统采用统一 schema、动态加载和热重载，新增工具后可被 Agent 自动发现；内部还实现了 create_skill / manage_skill，让主人可以通过对话要求机器人为自己开发或修复工具。为降低成本和提升稳定性，我还加入了规则预路由、任务去重缓存、长参数清洗、媒体上下文追踪、定时任务防误报、多 Agent 分工和工具执行统计。项目已在本机以 genesis-agent 和 genesis-web 两个 PM2 进程运行，既能在 QQ 中服务真实对话，也能通过 Web 面板查看运行状态、工具统计、配置和调度任务。
